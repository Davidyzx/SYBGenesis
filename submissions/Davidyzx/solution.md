## 2. 设计目标与「不做什么」
 
*（含你刻意选择「不游戏化」的边界）*
 
### 不做什么
 
大多数团队的"游戏化"之所以最终变成更精致的 KPI 牢笼，根源都是同一个错误：把游戏的表层符号:积分、勋章、排行榜...直接套在原有的绩效逻辑上。分数一旦可累加、可比较，人就会开始"打分数"而不是"做事"。所以在设计任何具体机制之前，我先制定好边界：
 
- **不做经验值/积分系统**——任何单一累加的数字，最终都会长成新的排行榜；
- **不做公开排名**——排名天然制造比较，和"不把人当资源池"直接冲突；
- **不做打卡型任务清单**（如"本周完成 N 个 PR 算达标"）——这类设计只会诱导人刷数量；
- **不做强制参与的赛季/竞赛机制**——SYB 是异步远程团队，强制的实时竞争节奏本身就和"异步"的工作方式相悖。
### 设计目标
 
**目标一：让能力而非资历成为解锁权限的主要凭证**
空洞骑士的世界不靠"打怪攒经验"打开新区域，而是靠获得具体能力（冲刺、蹬墙跳）解锁地图上原本走不通的路。抽象来看，这是一种把"进阶"和"时间沉淀"彻底解耦的设计：你能走多远，只取决于你已证明的具体能力，无关你在这里待了多久。翻译到 SYB：能接触更大项目、更高权限的系统，不该靠资历或职级，而应靠具体的、可验证的能力证明（比如独立完成过一次高质量的 code review、独立带过一次项目）。
 
**目标二：让失败的代价局限在"这一次尝试"，不累积成对人的长期评价**
死亡细胞里，每一局死亡都会清空局内资源，但解锁的"蓝图"永久保留，下一局能直接调用。这解决的核心问题是：失败之所以让人不敢冒险，不是失败本身，而是失败的代价会累积成别人对你能力的固定印象。死亡细胞把"这一次的结果"和"永久的成长"彻底拆开——局内清零，局外只增不减。翻译到 SYB：一次实验失败了，"局内"的东西（没达到的目标、具体成绩）不追责、不计入长期评价；但"局外"沉淀下来的东西——踩过的坑、复盘文档、验证过行不通的路径——变成团队共享的"蓝图"，永久留存。
 
**目标三：让"被认可"保持二元、不可比较**
宝可梦的道馆徽章证明你打过这一关，徽章之间没有相互比较，只有"有没有"，不存在"谁的徽章更闪"。翻译到 SYB：能力认证应该是"通过/未通过"式的里程碑，而不是可累加、可排名的分数——这是一种刻意选择的低分辨率，专门用来防止认可本身异化成新的 KPI。
 
 
## 3. 核心机制
 
*（借鉴了什么原理 → 翻译成什么 → 为什么适合 SYB）*
 
### 机制一：能力图谱——对应"能力而非资历"
 
**借鉴的原理**：空洞骑士靠具体能力（冲刺、蹬墙跳）解锁地图区域，而非等级数值；塞尔达的神庙彼此低耦合，你可以任意顺序去做，每个只教一件事，做完就是做完。两者共同的底层逻辑是：进阶是能力驱动、非线性的，不是时间驱动、线性的。
 
**翻译成什么**：一份公开的"能力图谱"——不是技能树式的必须按顺序解锁，而是像神庙一样，散布在地图上，谁都可以从任意一个开始。每个节点对应一项具体、可验证的能力（比如"独立带过一次生产事故复盘"、"独立 review 过一次跨模块 PR 并被作者采纳修改建议"）。完成即解锁，解锁后自动获得对应范围的项目/仓库访问权限，不需要人为审批。
 
**为什么适合 SYB**：远程异步团队没有"办公室里混脸熟"这条隐性晋升通道，权限分配必须靠可远程验证的证据，而不是"谁跟老板处得好"。能力图谱把这个证据结构化、公开化，天然契合"不信资历信 diff"的价值观。
 
### 机制二：蓝图库——对应"失败局内清零，局外永久"
 
**借鉴的原理**：死亡细胞的局内资源清零、局外蓝图永久保留。
 
**翻译成什么**：任何项目/实验结束（无论成败），发起人写一份"蓝图卡"——不是绩效复盘，而是纯粹面向后来者的知识资产：踩过的坑、验证过行不通的方案、可复用的临时工具/脚本。蓝图卡进入一个公开可搜索的仓库（比如 GitHub Discussions），不署名评价、只署名贡献——即卡片上不会写"这次项目没达标"，只会写"这条路走不通，原因是 X"。
 
**为什么适合 SYB**：异步团队最大的浪费是"教训不可传递"——一个人踩过的坑，因为没有面对面复盘的机会，很容易被下一个人重复踩。蓝图库把这个损耗降到最低。
 
### 机制三：徽章墙——对应"认可保持二元、不可比较"
 
**借鉴的原理**：宝可梦道馆徽章，通过/未通过的二元状态，徽章之间不可比较、不叠加成分数。
 
**翻译成什么**：把能力图谱上解锁的节点，可视化成个人主页上的一面"徽章墙"——只显示"有没有"，不显示"第几个拿到"、不显示排名、不做任何形式的加总或排序。徽章墙的作用仅仅是让别人快速判断"这个人证明过什么"，用于组队/授权决策，而不是用于横向比较优劣。
 
**为什么适合 SYB**：这是对"不做排行榜"这条边界最直接的落地——认可这件事必须存在（否则能力图谱无从谈起），但认可的呈现形式决定了它会不会变成攀比工具。二元、不可加总的展示方式，从界面层面就掐死了排行榜化的可能。
 
### 机制四：属性阵型——对应宝可梦的团队互补逻辑
 
**借鉴的原理**：宝可梦里没有一只"全属性最强"的宝可梦——火系克草系、草系克水系、水系克火系，循环克制。真正决定胜负的不是单体数值最高的那只，而是队伍六只的属性是否互补、能不能覆盖对手的弱点。堆一队清一色"最强"的同属性宝可梦，反而容易被针对性克制。
 
**抽象原理**：价值不是单轴的"从强到弱"排序，而是情境性、关系性的——一个成员值不值钱，取决于他和团队里其他人配合起来能不能补上当前的缺口，而不是他个人的绝对分数有多高。
 
**翻译成什么**：给每个人贴的不是"能力等级"，而是"属性标签"——比如"擅长根因排查"、"擅长架构设计"、"擅长文档/知识传递"、"擅长跨团队协调"，这些标签之间没有高低之分，只有类别之分。组建新项目团队时，机制上强制要求属性互补（比如不能同一个属性标签重复超过 N 人），而不是优先拉"综合评分最高"的人。这样"谁来做这个项目"这个问题，天然从"比谁更强"变成"谁能补上这块拼图"。
 
**为什么适合 SYB**：呼应"不把人当资源池"——资源池思维是把人当成同质化的算力单位、谁产出高就多用谁；属性阵型思维是承认每个人有不可替代的具体专长，组队本身就是对多元价值的制度化承认。
 
### 机制五：架势与复起——对应武士零的攻防节奏与"二次机会"
 
**借鉴的原理**：武士零的战斗核心不是靠消耗对方血条的"磨"，而是靠精准弹反、主动进攻打破对方的"架势条"——一次精准的判断，价值远高于反复的试探消耗。同时，死亡在这个游戏里不是终点：主角天生带着"复活"能力，死一次可以原地满血复起、立刻重新投入战斗，这个二次机会是写进系统底层的，不是外部给的怜悯。
 
**抽象原理**：（1）奖励精准和主动出手，而不是奖励耗时长、次数多；（2）"允许失败重来"不该是事后特批的例外，而应该是架构本身自带的保障——你不需要向谁申请，重来这件事本身就是规则的一部分。
 
**翻译成什么**：
- **架势评价**：一次关键的、精准的贡献（比如一次一针见血的根因定位、一次帮项目扭转方向的建议）在认定价值时，权重应高于单纯的"投入时长/产出数量"——直接对应"不信工时信产出"。
- **复起权**：任何提案/项目被否决或严重受挫后，发起人自动拥有一次无需特批、无门槛的重新提交机会（比如带着蓝图库里的教训，改一版再来），不需要走"申请二次机会"的流程，因为这次机会本身就已经写在规则里，不是谁的施舍。
**为什么适合 SYB**：这条机制是对"允许失败"这句口号最硬的落地方式——很多团队嘴上说允许失败，实际操作里"再申请一次"要看脸色、要被特批。把复起权写死在系统规则里，才能把"允许失败"从态度变成结构性保障。
 
### 小结
 
五个机制可以分成两类：**能力图谱、属性阵型、徽章墙**负责"怎么认定和呈现价值"；**蓝图库、复起权**负责"怎么处理失败和二次机会"。
 
## 4. 反馈回路
 
*（至少一条完整回路图 + 一个可被钻的空子 + 一个 18 个月后的崩点）*
 
### 完整回路：从"一次贡献"到"下一次贡献"
 
以一个具体场景走一遍这条闭环：
 
**1 行为发生**——A在一个项目里完成了一次高质量的根因排查，帮团队避免了一次更大的生产事故。
 
**2 被识别**——这次贡献被记录进能力图谱对应的节点（比如"独立完成一次生产事故根因定位"），因为是"架势式"的关键贡献而非单纯耗时长，这里适用机制五的架势评价权重——精准的一次判断，比反复试探的十次投入被更快识别。
 
**3 被反馈/激励**——能力图谱节点解锁 → 徽章墙上新增一枚徽章（二元展示，不加总、不排名）→ 属性阵型里，这个人在"根因排查"这个属性标签下的可见度提高。
 
**4 影响下一次行为**——下一个需要"根因排查"能力互补的项目组队时，机制四的属性阵型强制互补规则会优先把这个人纳入候选池，他获得参与更高价值项目的机会。项目结束后（无论成败），产出的经验写成蓝图卡进入蓝图库，供团队里下一个遇到类似问题的人调用——形成对其他人下一次行为的影响，回路由此从个体循环扩展为团队循环。
 
这条回路的关键特征是：从始至终没有一个可累加、可排名的数字——解锁是二元的，徽章是二元的，蓝图库是知识而非评分。整条闭环靠"能力→机会→再产出→再沉淀"驱动，而不是靠"分数→更高分数"驱动。
 
### 可钻的空子
 
最大的漏洞在第2步"被识别"这个环节：识别机制天然偏向可见、易归因给单一个人的贡献（比如"一次漂亮的根因定位"），而对不可见、难以归因、发生在幕后的贡献结构性地不友好——比如长期维护一个没人愿意碰的老旧模块、耐心带一个新人、持续做琐碎但必要的文档维护。
 
这会催生一种"表演型精算"策略：聪明人会学会挑选容易被看见、容易被单独归因的任务去做，主动避开那些价值真实存在但很难被识别成"一次关键贡献"的脏活累活。久而久之，能力图谱记录的不是"谁最有用"，而是"谁最会挑任务"。
 
### 18 个月后最可能先崩的地方
 
我认为最先崩的不是某个机制本身，而是属性阵型的标签固化：一开始"属性标签"是描述性的（这个人擅长什么），但随着能力图谱和徽章墙运行 18 个月，标签会逐渐变成入场券——组队时系统按标签匹配互补，会导致每个人被越来越频繁地分配到自己"已经有徽章"的领域，而很少被分配去尝试新领域。
 
结果是：本意是"承认多元价值"的属性阵型，运行足够久之后，会反过来把每个人焊死在自己最初被认定的那个标签上——这和空洞骑士"能力驱动而非固化身份"的初衷正好相反，变成了一种更隐蔽的种姓制度：不是用工龄分层，而是用"你 18 个月前被贴上的第一个标签"分层。这也是我在第 5 节设计"标签定期重申请"机制的原因。

## 5. 冷启动与运营
 
*（谁做、多久、用什么工具、怎么和现有流程接）*
 
### 谁维护
 
不设专职岗位——SYB 是几十人量级团队，专门养一个"游戏机制运营者"违背"高效"和"极致"的价值观本身。改为轮值召集人制度：从已经拿到至少 3 枚徽章的成员里，每季度轮换 1 人担任"图谱管理员"，职责只有三件事：审核能力图谱新节点的提案是否合理、仲裁边界模糊的能力认定、每季度汇总一次蓝图库做归档整理。这个角色本身也进入能力图谱，成为可解锁的一个节点——用机制本身激励人愿意去维护机制。
 
### 多久上线、怎么迭代
 
不建议一次性铺开五个机制。冷启动分三步、每步间隔约 6 周：
 
1. **第 1-6 周**：先上线蓝图库——因为它成本最低（只是一个公开仓库+一个模板），且能立刻产生可见价值（沉淀踩过的坑），用来验证团队对"公开写复盘、不追责"这件事的接受度。
2. **第 7-12 周**：上线能力图谱和徽章墙——需要先由轮值召集人和团队一起，共创出第一批 10-15 个能力节点（不宜一开始就铺几十个，容易变成形式主义清单）。
3. **第 13 周起**：上线属性阵型，因为它依赖前两个机制已经积累了真实数据（标签得从已发生的贡献里提炼，而不是靠自我申报）。
### 用什么工具
 
刻意不新造系统，全部用团队已经在用的 GitHub 生态承接：
 
- **能力图谱** → 一个仓库里的 `atlas.md`（结构化列表，PR 方式提议新节点，通过合并即生效）
- **徽章墙** → GitHub Profile README + 一个自动化脚本（每次 atlas.md 有新记录，脚本更新对应人的徽章展示）
- **蓝图库** → GitHub Discussions 或 `vault/` 目录，用 Issue 模板规范"蓝图卡"格式
- **属性阵型** → 一个简单的标签库（比如仓库里的 `CODEOWNERS` 风格文件），组队时人工参考，不做成强制自动分配的黑箱算法——算法自动分配容易演变成不透明的"系统说了算"，人工参考+公开标签更符合"开放"的价值观。
### 怎么和现有流程接
 
这五个机制并不是外挂在招聘/协作流程之外的新东西，而是同一套"fork→PR→互评"逻辑的自然延伸：
 
- 现在这道笔试题本身，就是能力图谱的第一个节点雏形——"独立完成一次开放式系统设计并通过同行互评"，完全可以直接收进正式上线后的 atlas.md；
- code review 文化本身就是"识别贡献"的天然入口——不需要额外的举报或申报机制，图谱管理员从常规 PR review 记录里就能找到值得记录的节点候选；
- 互评环节可以直接对应属性阵型里的"擅长评审/把关"这个标签的认定依据。
### 补丁：标签定期重申请（回应第 4 节的崩点预判）
 
针对第 4 节预判的"18 个月后标签固化"问题，从冷启动阶段就写死一条规则：属性标签每两个季度自动进入一次"开放期"——任何人可以在这两周内主动申请调整/新增自己的标签，且系统不保留"最初标签"的历史权重，新旧标签一视同仁。这条规则的目的很直接：不让"你第一次被贴的标签"变成永久身份，把重新定义自己的成本降到最低。
 
## 6. 我注入的价值观
 
*（每条价值观 ↔ 对应的具体机制）*
 
**价值观一：只信证据，不信自述**
能力图谱的每一个节点，认定依据只能是可公开验证的产出（PR、复盘文档、代码），不接受自我陈述式的"我觉得我擅长 X"。徽章墙同理——只展示"证明过什么"，不展示"自称什么"。
 
**价值观二：允许试错，但试错的代价不能累积成对人的标签**
蓝图库的局内清零/局外沉淀设计（机制二），以及复起权的无门槛重新提交（机制五）。
 
**价值观三：多元价值不可用单一尺子排序**
属性阵型的团队互补逻辑（机制四）、徽章墙的二元不可加总展示（机制三）。这条价值观回应的是第 2 节划的边界——不做排行榜——把"不排名"从一句消极的"不做什么"，落成"用什么代替排名"的积极设计。
 
**价值观四：规则本身要保持诚实，不能靠隐藏机制运作**
呼应"诚信→开放"这条价值观链的前两环。落点：属性阵型刻意不用黑箱算法自动分配（第 5 节），标签开放期公开透明、任何人可查（第 5 节的补丁机制）。

# 笔试题：把团队当作一款游戏来设计 —— 为 SYB 定义它的「游戏规则」

> **方向**：产品经理（开放式系统设计）
> **交付物**：一份机制设计文档（Markdown）
> **建议耗时**：8–16 小时（须在 fork 后 72 小时内提交 PR）
> **关联岗位**：[产品经理 · 武汉](http://211.159.165.135:12345/who-we-want/)

---

## 背景：欢迎来到创世纪

你正在参与 **Project Genesis（创世纪计划）**——SYB 想从 0 到 1，开创一种全新的组织形态。

你已经读过 [SYB 的招聘站](http://211.159.165.135:12345)：我们是一群散在不同城市、用远程协作造东西的研发者；不信资历信 diff，不信工时信产出，不信 PPT 信跑得起来的代码；信奉一条价值观链条——**诚信 → 开放 → 高效 → 极致 → 温度**。

一个组织能运转，靠的是一套「规则系统」：它决定谁能加入、谁说了算、做完一件好事怎么被看见、做完一件蠢事怎么被纠偏、人和人怎么协作、有人掉队怎么办。这套规则可以靠直觉和惯性长出来，也可以被**主动设计**。

**创世纪计划的第一个任务，就是写下这套最初的规则。而规则架构师，是你。**

## 你的核心任务

> **你是 Project Genesis 的规则架构师。请用「游戏的思维方式」，为 SYB 设计一套完整的团队运作机制——为这个新组织写下它最初的规则。**

「游戏的思维方式」**不是**给现有流程贴上「积分、勋章、排行榜」的皮肤——那是最低级的游戏化，我们没兴趣。我们想看到的是：你能不能从游戏几十年沉淀下来的**底层设计原理**里，提炼出可迁移到「一个真实运转的远程研发团队」的东西——目标系统、反馈回路、心流曲线、难度匹配、协作与博弈结构、经济系统的可持续性、玩家类型差异、意义的建构……

然后，把它们落到一套**真的能在 SYB 跑起来**的机制上。

## 你必须正面回应的五件事

这是开放题，没有标准答案，但下面五个问题**每一个**你都得给出明确立场和论证——端水的、四平八稳的、面面俱到却没主张的，在我们这儿等于没答。

### 1. 翻译，而不是搬运
你具体借鉴了哪几条游戏设计原理？请一条一条说清楚：它在游戏里本来是干嘛的、你把它翻译成了 SYB 里的什么机制、为什么这条原理适合「远程、扁平、异步、只看产出」的我们。我们要的是迁移的**精度**，不是术语的密度。

### 2. 系统自洽，而不是零件堆砌
机制之间会相互作用。你的设计里，至少要画/讲清楚一条**完整的反馈回路**（从某个行为发生，到它被识别、被反馈、被激励或纠偏，再到这个反馈如何影响下一个行为）。并且请预测一个**二阶后果**：聪明人会怎么钻你这套规则的空子？它运行 18 个月后，哪里最可能先崩？

### 3. 直面最大的矛盾
我们说「只看 diff、不为 KPI 而 KPI、绝不把人当耗材/资源池」。而游戏化天然倾向于把行为量化、计分、排名。**请直面这个张力**：你的设计里，哪些地方是你**刻意选择「不游戏化」**的？你凭什么相信它不会异化成一套更精致的 KPI 牢笼？—— 这是本题的灵魂，我们重点看你怎么答。

### 4. 落地，而不是科幻
SYB 是个几十人量级、远程、靠 GitHub / 文档 / 异步沟通运转的团队。请讲清楚：这套机制**冷启动**怎么开始？谁维护、维护成本是多少、用哪些现成工具？它怎么和我们「fork→PR→互评」的招聘、「code review」的质量文化对接？说不出落地路径的设计，在我们看来就是 PPT。

### 5. 埋入你的价值观
题面要求**尽可能包含你认可的价值观**。请明确列出 2–4 条你最看重的、你想注入这套机制的价值观，并指出：它们分别落在了哪个具体机制上（如果一条价值观找不到它对应的机制，那它就是口号）。我们想看你是谁。

## 交付物

一份**机制设计文档**（Markdown，不要 PPT——我们文档优先）。建议结构如下，但**结构本身也是考察项**：你可以推翻它、重构它，只要你能说明理由。

```
1. 我读懂的 SYB（3 句话以内，证明你不是泛泛而谈）
2. 设计目标与「不做什么」（含刻意不游戏化的边界）
3. 核心机制（借鉴了什么原理 → 翻译成什么 → 为什么）
4. 反馈回路（至少一条完整回路 + 一个可钻的空子 + 一个 18 个月后的崩点）
5. 冷启动与运营（谁做、多久、用什么工具、怎么和现有流程接）
6. 我注入的价值观（每条价值观 ↔ 对应的具体机制）
7. 设计日志（我推翻过哪些方案？我采纳或拒绝了哪些建议——含 AI 的？我留下的遗憾是什么？）
```

**篇幅**：3000–6000 字为佳。我们看重密度，不看重字数。配图（回路图、技能树、矩阵）鼓励，但每张图必须承载文字替代不了的信息。
